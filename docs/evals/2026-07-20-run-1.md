## Quality findings

1. **All five planted flaws surfaced.** The four targeted ones (N+1, missing
   policy, fat controller, missing tests) were each caught by their case, and
   the fifth — mass assignment — was surfaced **unprompted in three separate
   cases**. The `policy` run even caught the subtle variant: `$request->all()
   + ['user_id' => …]` lets a request-supplied `user_id` win because PHP's `+`
   gives the left operand precedence — a real privilege-escalation hole — and
   fixed it with explicit attributes.
2. **Fix quality was idiomatic.** `withCount` for the comment counts,
   domain-namespaced Action class, owner-only policy limited to
   `title`/`body`. Nothing needed `laravel-simplifier`-style cleanup.
3. **Parallelism worked where designed**: the `policy` case ran qa-engineer
   and security-engineer concurrently, as the command intends.

## Speed findings (the levers for the next pass)

Total wall-clock: **45 minutes for four single-command asks.** Where it went:

1. **Dynamic verification against an unrunnable app.** The fixture ships no
   `vendor/`, yet agents repeatedly tried to *run* it — performance-engineer
   spent 303s on "verify posts.index N+1 counts" it could never execute, and
   qa-engineer's 309–493s test stages include failed `artisan test` attempts.
   **Lever:** agents should detect a missing `vendor/` (or failed bootstrap)
   in one probe and fall back to static analysis immediately, stating so in
   the return instead of retry-flailing. This is the same failure family as
   the Sail feedback that produced `enforce-sail.sh`.
2. **Main-thread orchestration overhead is large and uneven**: total minus
   subagent time ≈ 82s (`n-plus-one`), ~400s (`policy`), ~320s (`action`),
   ~135s (`tests`). The heavy cases re-read the same fixture files the
   subagents also read. **Lever:** the stack-snapshot handoff exists but the
   *file-level* context (controller path, routes, model) is still re-derived
   per stage — briefs should carry the 3–4 load-bearing paths explicitly
   (commands already say this; the returns suggest it isn't consistently
   happening — worth a taught-rule or stronger command wording).
3. **qa-engineer is the consistent time/token hog** (309s, 493s, 83–86k
   tokens). Partly finding #1 (unrunnable test loops), partly writing more
   test cases than the brief asked for. **Lever:** re-check its model tier and
   add "test the brief's scenarios, list further scenarios as NEXT" wording
   before considering tier changes.
4. **Every agent event is emitted twice.** Each `start`/`end` appears
   duplicated (same `sid`, `ts`, payload) in `agents-board.jsonl` — so the
   `/board` dashboard double-counts runs. Durations/tokens are per-event and
   unaffected, but lanes render twice. **Lever:** dedupe by
   `(sid, ev, agent, ts)` in `emit-agent-events.sh` or in `board.html`'s
   reducer; diagnose whether the hook fires twice per matcher or the merged
   settings carry two entries.
