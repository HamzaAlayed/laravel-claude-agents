## Lever scorecard (the three 1.16.0 levers, verified)

1. **qa scope rule: worked.** The `tests` case — qa-engineer's worst stage —
   dropped **448s → 130s and 108.6k → 50k tokens (−54%)**, despite parallel
   contention pushing every other number up. The `action` case's qa stage
   stayed heavy (663s/109k), but that stage writes tests for a brand-new
   Action class, so the brief itself is big; no clean regression signal.
2. **Static-mode detection: no flailing, but agents choose a third path.**
   Nobody retry-flailed against the vendor-less app (run 2's failure noise is
   gone). Instead, the write-cases ran `composer install` and made the app
   real — `action` finished with `14 passed, 1 skipped`, `policy` with
   `16 tests, 0 failures`. Slower than static mode, but the verification is
   genuine; keeping it.
3. **Event dedupe: race found — the fix only worked when the twin hooks
   serialized.** Run 3's feeds still showed same-second duplicate lines. The
   two registrations run *concurrently*: both read the feed's last line before
   either appends, so the compare never fires. Fixed post-run with an atomic
   `mkdir` lock in `emit-agent-events.sh` (stale locks stolen after ~2s,
   fail-open) + a concurrent-twin guardrails test (#78).

## New findings

1. **Parallel mode trades measurement for wall-clock — and barely won.**
   Wall-clock was ~40 min vs run 2's ~44 min sequential, because four
   concurrent sessions (each spawning subagents) contend for the same API
   limits and CPU: `n-plus-one` inflated 96s → 619s (agent stage itself only
   161s). Documented in `tests/eval/README.md`: **parallel for pass/fail
   smoke, sequential for timing numbers.**
2. **The watchdog drifted 601s past its cap.** `policy` "timed out" at 2401s
   against an 1800s limit — the watchdog counted `sleep 5`s instead of
   reading a clock, and drifts badly under load. Fixed: wall-clock deadline +
   TERM→KILL escalation after 30s grace.
3. **`policy` went fully async and disappeared from the board.** Its
   coordinator launched all three subagents in background mode; their `end`
   events carry `status:"async_launched"` with `ms:null, tokens:null`, and no
   completion event ever lands. Correct per Claude Code's hook payloads (the
   PostToolUse fires at *launch*), but it means async stages are invisible to
   the board and to eval timing. Future lever: a SubagentStop-based emitter,
   if hooks expose enough identity there.
4. **Fixture unrealism taxed every write case.** Three agents independently
   bootstrapped around the same gaps vs a real Laravel 13 skeleton: no
   `mockery/mockery` in require-dev (suite can't even boot — every write case
   added it and flagged the dependency change for approval), no
   `.env.example`, no `storage/`+`bootstrap/cache/` dirs, and a `posts_total`
   stats row no seeder creates. All fixed in the fixture after this run —
   write cases should get minutes faster and stop flagging phantom approvals.
   (One hallucination noted for the record: `policy`'s report claimed the
   `followers`/`site_stats` migrations don't exist; they do.)
